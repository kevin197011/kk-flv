# PRD: FLV 代理拉流与前台播放

## 1. 背景与目标

### 1.1 背景
- 存在仅内网或后端可访问的 FLV 流地址（如 `http://10.170.96.153:8081/LesA/113503.flv`）。
- 前台页面无法直连该地址（跨域、网络隔离或安全策略）。
- 需要在前台输入 FLV 地址，由后端拉取数据并转发给前台，用于播放测试。

### 1.2 目标
- 用户在前台输入一个 HTTP(S) FLV 地址。
- 后端作为代理从该地址拉取 FLV 数据并流式返回给前台。
- 前台使用标准视频/FLV 播放能力进行播放测试。

### 1.3 非目标（本阶段）
- 不实现用户鉴权、流权限控制（可后续扩展）。
- 不实现录制、转码、多路流管理。
- 不保证超低延迟，以满足“拉流 + 播放测试”为主。

## 2. 用户故事与场景

| 角色 | 故事 | 验收标准 |
|------|------|----------|
| 测试/运维 | 输入内网 FLV 地址并在页面播放 | 输入 URL → 点击播放 → 视频可正常播放 |

## 3. 功能需求

### 3.1 前台
- **地址输入**：提供输入框，用于输入 FLV 的 HTTP(S) 地址（如 `http://10.170.96.153:8081/LesA/113503.flv`）。
- **播放触发**：提供“播放”或“加载”操作，将当前输入的地址提交给后端代理接口。
- **播放器**：使用可播放 FLV 的播放器（如 flv.js + `<video>`）播放后端返回的流。
- **状态与错误**：展示加载中、播放中、错误等状态；拉流或播放失败时给出可读错误信息。

### 3.2 后端
- **代理接口**：提供 REST 接口，接收前台传来的 FLV 地址（或通过 query/body 传递）。
- **拉流与转发**：后端使用 HTTP Client 请求该地址，将响应体以流式方式（Stream）转发给前台，并设置合适的 `Content-Type`（如 `video/x-flv`）。
- **安全与校验**：仅允许 HTTP/HTTPS 的 FLV 地址；可对 URL 做白名单或域名/IP 校验（按需配置）；避免 SSRF 风险（禁止 file://、内网敏感端口等，按策略配置）。

### 3.3 约束
- 仅支持 GET 方式的拉流（源为 GET 即可）；若后续需 POST 或 Header 透传，可再扩展。
- 流式转发，避免将整文件读入内存再输出；控制超时与最大 body 大小（可选），防止长时间占用连接。

## 4. 接口设计

### 4.1 代理播放接口（推荐）

**方式 A：GET + query**

```
GET /api/v1/flv/proxy?url=<encoded_flv_url>
```

- `url`: 经 URL 编码的 FLV 地址（仅允许 http/https）。
- **Response**: 流式二进制，`Content-Type: video/x-flv`，直接作为播放器源。

**方式 B：POST + JSON body（可选，便于后续加 Header/参数）**

```
POST /api/v1/flv/proxy
Content-Type: application/json

{ "url": "http://10.170.96.153:8081/LesA/113503.flv" }
```

- **Response**: 同上，流式返回 FLV。

本 PRD 优先实现 **方式 A**，实现简单、易与 `<video src="...">` 或 flv.js 的 URL 对接。

### 4.2 错误响应
- `400`: URL 缺失、非法或非 http(s)。
- `502` / `504`: 上游拉流超时或不可达。
- `500`: 服务端内部错误。
- 错误体建议 JSON：`{ "code": "xxx", "message": "..." }`。

## 5. 前端交互流程

1. 用户打开“FLV 代理播放”页面。
2. 在输入框中输入 FLV 地址（如 `http://10.170.96.153:8081/LesA/113503.flv`）。
3. 点击“播放”或“加载”。
4. 前端将地址编码后请求后端代理接口（如 `GET /api/v1/flv/proxy?url=...`），并将该接口 URL 作为播放器源（或通过 flv.js 拉取该 URL）。
5. 后端从目标地址拉流并流式返回，前端播放器开始播放。
6. 若失败：显示错误信息（如“无法拉取流”或后端返回的 message），不阻塞用户修改地址后重试。

---

## 6. 非功能需求

- **安全**：URL 校验、禁止非 http(s)、SSRF 防护（禁止访问内网敏感端口或可配置白名单）。
- **可观测性**：后端对代理请求打日志（不记录完整 URL 若含敏感信息时可只记录 host/path）。
- **超时**：后端对上游请求设置合理连接/读超时（如 10s 连接、60s 读），避免挂死。
- **部署**：后端、前端可 Docker Compose 运行，便于本地与测试环境一致。

---

## 7. 技术要点（实现参考）

- **后端（Go）**：使用 `net/http` 请求用户提供的 URL，校验 scheme 与 host；使用 `io.Copy` 或类似方式将 `resp.Body` 流式写入 `ResponseWriter`，并设置 `Content-Type: video/x-flv`。
- **前端**：可使用 **flv.js** 将 FLV 流解码后喂给 `<video>`，或支持 FLV 的其它播放器；播放源为当前站点下的代理接口 URL，避免跨域。
- **CORS**：若前后端同域则无需额外 CORS；若分离部署，后端需对前端 Origin 放行。

- [ ] 前台可输入 FLV 地址并点击播放。

## 9. 如何运行

### 本地（前后端分离）
```bash
# 终端 1：启动后端
cd backend && go run ./cmd/api
# 默认 http://localhost:8080

# 终端 2：启动前端
cd frontend && npm install && npm run dev
# 默认 http://localhost:3000，/api 已代理到后端
```
浏览器打开 http://localhost:3000，在输入框填入 FLV 地址（如 `http://10.170.96.153:8081/LesA/113503.flv`），点击「Play」即可。

### Docker Compose
```bash
docker compose up --build
```
后端：http://localhost:8080，前端：http://localhost:3000。前端通过环境变量 `API_PROXY_TARGET=http://backend:8080` 将 `/api` 代理到后端。
